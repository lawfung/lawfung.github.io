程式競賽debug小筆記

Bug 可以出現在任何地方，因此沒有泛用的 debug 流程，本文僅僅提出一些很粗淺的技巧。

常見的坑

  • 檢查題目數字範圍
    • 你陣列開的夠大嗎
    • 你的程式可以處理極端狀況嗎 (極大或是極小)
  • 你的運算過程中會 overflow 嗎
  • 你的運算過程中會發生無限迴圈（遞迴）嗎
  • 你寫的演算法真的是對的嗎

Warning

相較於以上兩種做法，開 Warning 其實是非常經濟實惠的作法，你要多做的事情就只有在編譯時多增加一些參數。
Warning 的意義在於，當你編譯時，有些寫法可以通過編譯器，但是實際上你卻不想要這樣寫，也就是說，一旦你這樣寫，其實你是寫錯的，但是編譯器卻沒有辦法偵測出來。比如說，宣告一個和全域變數同名的區域變數，你應該不會想要這樣做，但是有時候不小心還是會寫成這樣。但是，如果你開了 Warning ，這類實際上經常是寫錯的寫法就可以在編譯時跳出警告。

Warning 的種類有很多，以上的四種中

  • -Wall
  • -Wextra
  • -Wshadow
  • -Wconversion

前兩種其實是類似大禮包的東西，會一次開一大堆的 Warning 。
後兩者則各自有自己的功能，-Wshadow大概就是檢查同樣的名字重複宣告，而-Wconversion大概就是檢查有沒有變數被編譯器強制轉型，有興趣的人可以自行查閱。
如果沒有甚麼特別需求的人，照抄我這四種 Warning 應該就夠用了。
